===================================================== SCOPO DEL
DOCUMENTO =====================================================

Questo documento definisce l’architettura ufficiale della pipeline di
ingestione eventi del sistema LOGOS.

La pipeline collega le sorgenti di input esterne al ledger eventi
RAW_DATA garantendo:

• validazione dati • normalizzazione entità e progetti • scrittura
controllata nel ledger • logging del motore • scalabilità futura verso
DB esterni

===================================================== ARCHITETTURA
GENERALE =====================================================

EVENT PRODUCERS (Form / Voice / Manual / API) ↓ RAW_INPUT ↓
LOGOS_PROCESSOR (worker pipeline) ↓ RAW_DATA (Event Ledger) ↓ DATASET
ANALITICI ↓ DASHBOARD

===================================================== EVENT PRODUCERS
=====================================================

Le sorgenti di eventi possono essere:

• Google Form • inserimento manuale • input vocale / Siri Shortcut • API
esterne • import batch

Tutti gli eventi entrano nel sistema tramite RAW_INPUT.

===================================================== RAW_INPUT
=====================================================

RAW_INPUT è il buffer di ingestione eventi.

Contiene input grezzi provenienti dalle interfacce utente.

Schema colonne:

A Timestamp_Form B Project_Name C Tipo_Movimento D Valore E Data_Evento
F Causale G Riferimento_ID H Metodo_Pagamento I Note J Soggetto K Source
L Status

Status possibili:

NEW ENTITY_PENDING ERROR_PROJECT WRITTEN

===================================================== LOGOS_PROCESSOR
=====================================================

Il modulo LOGOS_PROCESSOR trasforma gli eventi RAW_INPUT in eventi
canonici del ledger.

Funzioni principali:

• validazione progetto • risoluzione entità • normalizzazione dati •
scrittura batch nel ledger • aggiornamento stato eventi • logging nel
SYSTEM_LOG

===================================================== CICLO DI VITA
EVENTO =====================================================

NEW ↓ VALIDATION ↓ ENTITY_CHECK ↓ WRITTEN

Possibili deviazioni:

ERROR_PROJECT ENTITY_PENDING

===================================================== MAPPING RAW_INPUT
→ RAW_DATA =====================================================

Project_Name → Project_ID Soggetto → Entity_ID

Campi derivati generati dal sistema:

ID_Movimento Timestamp Categoria_Derivata Unità standard

===================================================== RAW_DATA (EVENT
LEDGER) =====================================================

RAW_DATA rappresenta il registro eventi ufficiale.

Caratteristiche:

• append-only • eventi immutabili • tracciabilità completa •
compatibilità con dataset analitici

===================================================== SYSTEM_LOG
=====================================================

Il foglio SYSTEM_LOG registra la telemetria del motore.

Eventi principali:

INPUT_PROCESSOR_START INPUT_ERROR_PROJECT INPUT_ENTITY_PENDING
INPUT_WRITTEN INPUT_BATCH_COMPLETED

Questo consente monitoraggio e debugging del sistema.

===================================================== TRIGGER RUNTIME
=====================================================

Il processor è attivato tramite trigger time-based.

Configurazione suggerita:

Trigger: processRawInputV2()

Frequenza: 1 minuto

Limite sicurezza:

MAX_EVENTS_PER_RUN = 50

===================================================== SCALABILITÀ
=====================================================

L’architettura è progettata per essere portabile su infrastrutture
database future:

• Baserow • PostgreSQL • SQLite

La logica del processor rimane invariata mentre cambia solo il livello
di persistenza dati.
